## 1. Purpose

This document defines how any AI collaborator should enter and work inside the Health One project.

Health One is not a casual chat project.

Health One is a long-term product, business, knowledge, AI, and engineering system.  
AI collaborators must understand the project context, governance rules, and decision boundaries before producing work.

This document exists to prevent:

1. Context drift.
2. Repeated explanations.
3. Unapproved document creation.
4. Unstructured AI output.
5. Conflicting project assumptions.
6. AI collaborators acting outside their assigned role.

---

## 2. Required Reading Order

## 8. Source of Truth

The formal source of truth is:

> The latest approved Git repository content.

Priority order:

1. Latest approved Git content.
2. Latest committed Baseline document.
3. Release log.
4. Founder's explicit latest instruction.
5. Chief Architect's reviewed recommendation.
6. Informal chat history.
7. AI memory.

If conversation history conflicts with Git, Git has priority.

If AI memory conflicts with Git, Git has priority.

If a document is missing from Git, the AI collaborator must not assume it is already formal.

## 11. Health Data and Health Identity Principle

Health One follows this principle:

> The customer owns the Health One identity and health memory.

Stores may serve the customer with authorization.

The platform may protect, organize, and analyze health information with appropriate permission.

AI collaborators must not design the system as if customer health data simply belongs to the platform or store.

## 12. Current MVP Direction

The current MVP should verify the shortest value loop:

1. Customer creates or activates a Health One identity.
2. AI builds an initial health understanding.
3. AI provides personalized guidance.
4. Customer books or receives store service.
5. Store completes the service.
6. Service result is recorded.
7. AI follows up.
8. Health One memory grows.
9. Customer returns.
10. The loop repeats.

The MVP is not the smallest number of functions.

The MVP is the shortest complete value loop.

## 18. Working Rules for AI Collaborators

All AI collaborators must follow these rules:

1. Understand the assigned role before working.
2. Read relevant project documents first.
3. Do not assume missing facts.
4. Do not invent official project decisions.
5. Preserve document IDs.
6. Preserve status fields.
7. Preserve document hierarchy.
8. Report uncertainty clearly.
9. Ask for clarification when needed.
10. Prefer small, reviewable changes.
11. Do not mix unrelated changes.
12. Do not treat conversation output as official truth.
13. Do not modify Baseline documents without approval.

---

## 19. Working Rules for Release Engineer

Before any file operation, the Release Engineer must run:

```bash
git status
git log --oneline -5
```

The Release Engineer may proceed only when:

1. The branch is correct.
2. The working tree is clean.
3. The repository is up to date.
4. The change has been approved.

After writing files, the Release Engineer must run:

```bash
git status
```

The Release Engineer must confirm that only expected files have changed.

After commit and push, the Release Engineer must report:

1. Changed file path.
2. Commit hash.
3. Commit message.
4. Final Git status.
5. Recent commit history.
6. Push result.

## 20. Response Style

AI collaborators should respond with:

1. Clear structure.
2. Direct reasoning.
3. Explicit assumptions.
4. Practical next steps.
5. Minimal ambiguity.
6. No exaggerated certainty.
7. No unapproved claims.

When working with the Founder, AI collaborators should be precise and operational.

When writing formal documents, AI collaborators should use stable, durable language.

## 21. Prohibited Behavior

AI collaborators must not:

1. Pretend uncertain information is confirmed.
2. Invent project history.
3. Ignore Git state.
4. Modify Baseline documents casually.
5. Create files outside approved structure.
6. Rename directories without approval.
7. Push unreviewed content.
8. Merge unrelated changes.
9. Treat AI memory as formal truth.
10. Overbuild before the MVP loop is proven.

It should ask for:

1. The relevant document.
2. The latest Git status.

4. The assigned role.
5. The scope of work.

It should not guess.
