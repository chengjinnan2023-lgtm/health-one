## 1. Problem

### 当前状态

- `admin`（店长）和 `staff01`（健康管理师）登录后看到的页面完全相同。
- 两个角色可以执行完全相同的操作：新建客户、编辑档案、记录服务、创建随访。
- JWT payload 包含 `role` 字段，但前端完全没有使用它来区分页面或操作权限。

### 为什么现在需要解决

Pilot 阶段只有 1-2 名店员，角色不区分问题不大。但扩大试运行后：

- 店长需要看到全局视图（所有客户的随访完成率、谁还没有随访）
- 健康管理师只需看到自己负责的客户和任务
- 如果前台/运营人员也要使用系统（查看但不操作），需要只读权限

不做角色区分会导致：店长"淹没"在与店员相同的操作页面中，看不到全局；店员可能误操作不该操作的功能。

## 2. Roles

### 当前角色

| 角色 | 中文 | RFC-002 枚举值 | 当前人数 | 职责 |
|------|------|-------------|---------|------|
| Manager | 店长 | `店长` | 1 | 管理门店运营、查看全局数据、分配随访任务 |
| Health Advisor | 健康管理师 | `健康管理师` | 1-2 | 服务客户、记录服务、创建随访 |

### 建议新增角色

| 角色 | 中文 | 建议枚举值 | 人数 | 职责 |
|------|------|----------|------|------|
| Service Staff | 服务人员 | `服务人员` | 1-2 | 执行服务、记录反馈（不创建随访） |
| Observer | 观察者 | `观察者` | 1 | 只读查看客户数据（Pilot 观察员、投资人、Founder） |

**当前 RFC-002 §3.7 已有三个枚举值：** `店长`、`健康管理师`、`服务人员`。不需要修改模型——已有的枚举足够覆盖前三个角色。`观察者`可暂用 `服务人员` + 前端只读限制实现。

## 5. Recommended MVP Rule

### 最小差异化方案（建议 Sprint-4 实施）

**不要做完整的 RBAC 系统。只做两件事：**

1. **首页差异化：** 根据 JWT 中的 `role` 字段渲染不同的首页组件。
   - `店长` → 店长首页（全局概览 + 待处理随访）
   - `健康管理师` → 健康管理师首页（我的随访 + 我的服务）
   - 其他角色 → 健康管理师首页（默认）

2. **按钮/操作隐藏：** 在前端根据 `role` 渲染或隐藏特定按钮。
   - `店长`：全部可见
   - `健康管理师`：隐藏"归档客户"按钮
   - `服务人员`：隐藏"创建随访"按钮、"新建客户"按钮
   - `观察者`：全部只读（隐藏所有保存/创建按钮）

### 实施方式

```tsx
// 前端判断角色
const { staff } = useAuth();
const isManager = staff?.role === "店长";
const isHealthAdvisor = staff?.role === "健康管理师";

// 条件渲染
{isManager && <button>归档客户</button>}
{isManager && <ManagerDashboard />}
{isHealthAdvisor && <HealthAdvisorDashboard />}
```

**不修改后端权限逻辑。** 当前所有 API 端点从 JWT 中提取 staff_id，不区分角色——保持这个逻辑。角色差异仅在 UI 层面实现。后端权限控制（API 级别拒绝非店长的归档操作）在 Sprint-5+ 引入。

### 为什么最小化

- 当前只有 2-3 名店员，不需要复杂的权限系统
- 前端条件渲染足以满足当前角色区分需求
- 后端权限控制可以在需要时追加（不影响前端接口）
- Pilot 阶段重点验证"角色区分是否有用"，不做过度设计

## 6. Out of Scope

| 不做 | 原因 |
|------|------|
| 完整 RBAC 引擎（角色-权限-资源三级模型） | MVP 不需要——2-3 名店员 |
| 总部多级审批（店长审批 → 区域经理审批） | 单店 Pilot |
| 组织架构系统（部门、汇报线） | Sprint-5+ |
| 字段级权限（店长可看收入、健康管理师不可看） | 当前无敏感字段 |
| 动态权限配置页面（店长自己配权限） | 过度设计 |
| 操作审计日志（谁在什么时候做了什么） | Sprint-5+（Timeline 已部分覆盖） |
| JWT role 在后端 API 级别校验 | Sprint-5+（当前 API 通过 staff_id 鉴权足够） |

## 7. Recommendation

### 是否进入开发？

```
YES — 建议 Sprint-4 实施。
```

### 优先级

```
P1 — 在 Pilot 扩大试运行完成后、Sprint-4 AI 功能之前实施。
```

**理由：** 角色区分是"规模化"的前提，不是"闭环验证"的前提。Pilot 阶段可以不区分角色——现在验证的是"系统能不能跑"。但一旦进入多店员常态化使用，角色区分就成为必需。建议在 Sprint-4 的第一个 Phase 实施（与 DEV-030 Auth Wiring 类似——基础设施类改动，优先完成）。

### 先做页面差异，还是先做按钮权限控制？

**先做页面差异。** 理由：

1. 页面差异立即可见——店员打开系统就知道"这是我的工作台"
2. 按钮权限控制是防御性的——防止误操作，但在当前信任环境下（同店店员）发生概率低
3. 页面差异实施成本更低——只需根据 `role` 渲染不同首页组件

**实施顺序：**
1. Sprint-4 Phase 0：首页差异化（店长首页 vs 健康管理师首页）—— 0.5 天
2. Sprint-4 Phase 1：按钮/操作隐藏（角色条件渲染）—— 0.5 天
3. Sprint-5+：后端 API 级别 role 校验 —— 1 天

## 8. End of Document

ROLE-001 proposes minimal store role differentiation for post-Pilot implementation.

**Recommendation: P1, Sprint-4. Start with homepage split, add button permissions later. No RBAC engine.**
