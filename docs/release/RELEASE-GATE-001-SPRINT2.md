## 6. Known Limitations

### 6.1 Cannot Demo Full Closed Loop Without PostgreSQL

Identity creation, profile update, and timeline queries require a running PostgreSQL instance. The Store DB (SQLite) handles auth independently. A full end-to-end demo requires:
- PostgreSQL running locally or in CI
- Platform DB migration applied
- Both Platform API and frontend dev server running

### 6.2 Auth Is Implemented But Not Enforced on All Routes

The `get_current_staff` middleware exists but is not applied to identity/profile/timeline routes. The frontend sends JWT tokens correctly, but the backend doesn't reject unauthenticated requests. This is documented as AD-002 from QA-BATCH-003 and tracked for Sprint 3.

### 6.3 Single-Store Only

The MVP is designed for a single pilot store. Store DB uses a single SQLite file. Multi-store deployment (multiple SQLite files, one per store) is architected but not tested.

### 6.4 No Production Deployment

Per ADR-002 §3.6, deployment is manual (Nginx + systemd) and deferred. All code has been tested in development mode only. CI pipeline is configured but no deploy job exists.

### 6.5 Frontend Has No Backend Validation Feedback

Form validation is minimal (required fields checked client-side). Server-side validation errors (422) are displayed as generic error messages. The API returns structured Pydantic validation errors, but the frontend doesn't parse them for field-level feedback.

### 7.1 Prerequisites

```bash
# 1. PostgreSQL running with health_one_platform database
# 2. Python venv with dependencies installed
# 3. Node.js with frontend dependencies installed
```

### 7.2 Backend Setup

```bash
cd /path/to/health-one
source venv/bin/activate

# Run Platform DB migration (requires PostgreSQL)
cd health_one/platform && alembic upgrade head

# Run Store DB migration + seed
cd ../store && alembic upgrade head
python -m health_one.store.seed

# Start Platform API
uvicorn health_one.platform.main:app --reload --port 8000
```

```bash
cd frontend
```
