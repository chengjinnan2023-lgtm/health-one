## 1. Manager Objective

店长是门店运营的核心角色。在 Health One 体系中，店长的职责是：

1. **掌握全局** — 知道门店今天发生了什么、本周/本月趋势如何
2. **防止遗漏** — 确保没有客户的随访被忘记、没有服务被遗漏
3. **数据驱动** — 能导出数据、用数据做运营判断
4. **客户管理** — 管理客户生命周期（激活、归档、标签分类）

店长不需要亲自执行每一项服务，但需要对门店运营结果负责。

## 3. Daily Workflow Walkthrough

### 3.1 登录

```
店长打开 Health One
→ 输入 admin / health123
→ JWT 签发 role=店长
→ 前端判断 role === "店长" → 渲染 ManagerDashboard
→ 看到标题"店长工作台" + "张三，欢迎回来"
```

**当前体验：** 店长立即感知到"这是我的专属工作台"，而非通用的健康管理师页面。角色分化在登录后第一屏即生效。

### 3.2 看首页

```
店长工作台
├── 今日概览（3 卡片）
│   ├── 今日新增客户：2
│   ├── 今日服务：5
│   └── 待随访：4
├── 客户结构
│   ├── 总客户 12 | 已激活 8 | 待激活 2 | 已归档 2
│   └── 标签：肩颈(5) 老客户(3)
├── 最近服务（5 条）
│   ├── 张伟 — 健康舱 — 今天 ✓
│   └── 李娜 — 咨询 — 今天 …
├── 待随访（预览 10 条）
│   ├── 张伟 — 📞 电话 — 7/3
│   └── 李娜 — 💬 微信 — 7/4
│   └── [查看全部 →]
├── 📥 导出日报（3 按钮）
└── [客户管理] [运营统计]
```

**当前体验：** 一屏看到当日门店全貌。数据来自单一 API（`GET /api/dashboard/manager`），加载一次即完成。

**已知限制：**
- 今日概览 3 个数字均为计数，无对比（如"昨天 3 人"）
- 待随访预览仅显示 10 条，其余需点击"查看全部"
- dashboard API 返回的待随访最多查询 50 条 plan（limit 50）

### 3.3 看待跟进客户

```
店长点击"查看全部 →"
→ 进入 /follow-up-queue
→ GET /api/dashboard/follow-up-queue
→ 返回合并去重后的完整队列

待跟进客户
共 6 个待跟进客户 · 随访计划 4 人 · 标签标记 2 人

┌────────────────────────────────────────────┐
│ 张伟                    [已激活]            │
│ 📞 电话随访 · 计划 2026/07/03              │
│ 肩颈 · 老客户               [查看客户 →]    │
├────────────────────────────────────────────┤
│ 李娜                    [已激活]            │
│ 🏷 高意向 · 暂无随访计划                    │
│ 高意向 · 肩颈               [查看客户 →]    │
└────────────────────────────────────────────┘
```

**当前体验：** 店长可以看到所有待跟进的客户，包括有随访计划的和仅标签标记的。每条有清晰的信号来源（📞 随访 / 🏷 标签）、计划时间和操作入口。

**已知限制：**
- 队列是全店范围，未区分"店长负责的"与"健康管理师负责的"（当前无 staff-customer 归属）
- 同客户多个 pending plan 只显示最近一个
- 页面未标注"全店待跟进客户"

### 3.4 看运营统计

```
店长点击"运营统计"
→ 进入 /manager/stats
→ GET /api/dashboard/manager/stats?period=week（默认）

运营统计
2026-06-29 ~ 2026-07-05

[本周] [本月]  ← 可切换

┌────────┬────────┬────────┐
│ 新增客户│ 服务记录│ 完成随访│
│   3    │   12   │   2    │
└────────┴────────┴────────┘

客户结构
总12 | 已激活8 | 待激活2 | 已归档2

标签 Top 5
肩颈(5) 老客户(3) …
```

**当前体验：** 店长可以按周/月切换查看门店运营数据。3 个核心指标（新增客户、服务记录、完成随访）+ 客户结构 + 标签分布，覆盖基本运营判断需求。

**已知限制：**
- 无双周期对比（如"上周 vs 本周"）
- "完成随访数"使用 plan.updated_at 近似完成时间（无 dedicated completed_at 字段）
- 时区为 UTC（已知问题，Pilot 阶段影响小）

### 3.5 导出日报

```
店长在首页"导出日报"区域
→ 点击"导出今日客户 CSV"
→ fetch() 携带 JWT Bearer token
→ 后端返回 CSV（BOM + UTF-8 + 中文表头）
→ Blob 下载 → 浏览器自动保存为 customers-2026-06-30.csv

Excel 打开：
姓名    状态    标签           创建时间
张伟    已激活   肩颈、老客户    2026-06-30 10:30
```

**当前体验：** 一键导出 3 类 CSV（今日客户 / 今日服务 / 待随访）。中文可读，Excel 不乱码。文件名含日期，便于归档。

**已知限制：**
- 仅导出当天数据（无历史日期范围选择，但 API 支持 export_date 参数）
- followups CSV 忽略 export_date（始终返回全量 pending）
- 无"导出全部"按钮

## 9. Recommendation

### 9.1 当前店长端是否已经具备基本可用性？

```
YES — 店长可以独立完成"看全局 → 找问题 → 导出数据 → 管理客户"的完整日常闭环。
```

当前店长端有 4 个独立页面（首页、待跟进队列、运营统计、日报导出），覆盖了店长日常管理的核心场景。Pilot 门店店长可以用这些能力完成一天的管理工作。

### 9.2 店长最有价值的页面是哪个？

```
店长首页（ManagerDashboard）—— 一屏看全局，是所有管理操作的起点。
```

首页聚合了今日概览、客户结构、最近服务、待随访预览、导出入口和快捷入口。店长打开系统后的第一个决策（"今天该关注什么"）在这里完成。

**第二名是待跟进队列** —— 当店长需要具体跟进客户时，队列是"安全网"。

### 9.3 下一步对店长最值得做的功能是什么？

```
店员管理 > 更细统计 > AI 总结
```

**推荐优先级和理由：**

| 优先级 | 功能 | 理由 |
|--------|------|------|
| **P1 — 店员管理** | Staff CRUD + 列表页 + 工作量统计 | 店长管理的是"人"。没有店员管理，店长端只是"数据看板"而非"管理工具"。当前 admin/staff01/staff02 需手动管理 SQLite，扩大 Pilot 后会成为瓶颈 |
| **P2 — 更细统计** | 周期对比 + staff 维度 + 客户活跃度 | 在店员管理就位后，统计需要回答"谁做了多少"和"趋势如何" |
| **P3 — AI 总结** | 门店周报 / 客户洞察摘要 | AI 有价值，但在"人"的管理问题解决之前，AI 总结无法落地——店长首先需要知道"谁负责什么" |

**为什么店员管理是第一优先级：**

1. 当前 Pilot 只有 3 名店员，手动管理可接受。但扩大 Pilot（5+ 店员）后不可行
2. 随访任务分配（P1-3）依赖店员列表——没有 Staff CRUD，就没有"分配给谁"
3. 店员工作量统计（P1-2）依赖 staff 维度的数据——没有店员管理，就没有归属
4. 健康管理师的"我的客户"和"我的随访"依赖 staff-customer 归属——没有归属，角色分化只做了一半

**店员管理 MVP 建议范围（Sprint-5）：**
- 店员列表页（店长可见）：姓名、角色、状态
- 新增店员表单：姓名、角色、密码
- 编辑/禁用店员
- 不涉及：复杂权限配置、组织架构、审批流程
