## 1. Problem

### 当前状态

- 客户可以"激活"和"归档"，但归档后不可恢复（`POST /archive` 是单向操作）
- 客户没有任何分类标签——店员只能通过名字搜索
- 客户列表按时间倒序排列，无法按类型筛选
- Pilot #1 中，店员反馈"客户多了以后不知道怎么看"

### 门店真实需求

**场景 1：** 客户一年没来，店员不确定是"已经流失"还是"不需要服务"。如果删掉，万一又来了，历史记录丢失。如果不删，列表越来越长。

**场景 2：** 店长想做针对性的回访——"把所有'肩颈'关注的客户找出来，统一发一条回访消息"。目前只能逐个搜索。

**场景 3：** 店员想给客户打标——"这个客户对价格敏感"、"这个客户喜欢周末来"、"这个客户是朋友介绍来的"。这些信息目前只能写在备注里。

---

## 2. Recommendation

### 三个能力

| 能力 | 说明 |
|------|------|
| **Archive / Unarchive** | 归档不活跃客户，但可恢复。不永久删除。 |
| **Customer Tagging** | 店员可手动为客户添加/移除标签。 |
| **Tag Filter** | S1 搜索页支持按标签筛选客户。 |

### 核心原则

> 不做永久删除。不做自动标签。不做复杂规则引擎。只做门店日常运营需要的最小分类能力。

---

## 3. Why

### 不做永久删除

- Constitution §8：数据属于用户。删除客户 = 删除健康记录，违反可追溯原则。
- Timeline 是 append-only——删除客户意味着级联删除 Timeline，数据不可恢复。
- 门店可能误操作。"归档"是安全的——误归档可以恢复；误删除不可恢复。

### 做 Archive / Unarchive

- 当前已有 `POST /archive`，但缺少 `POST /unarchive`。补充即可。
- 归档后客户不出现在默认列表中（减少视觉噪音），但仍可通过"已归档"筛选找到。
- 归档客户的历史数据完整保留。

### 做最小 Tagging

- 标签是店员自己定义的、非结构化的分类方式。
- 不需要后端定义标签枚举——标签是自由文本，由门店自己管理。
- 提高查找和回访效率：按标签筛选 → 批量查看 → 针对性服务。

---

## 4. MVP Scope

### 包含

| 功能 | API | 前端 |
|------|-----|------|
| 客户归档 | `POST /api/identities/{id}/archive`（已有） | S2 按钮（已有） |
| 客户恢复 | `POST /api/identities/{id}/unarchive`（新增） | S2 按钮：已归档状态显示"恢复" |
| 为客户添加标签 | `PATCH /api/identities/{id}` 新增 `tags: string[]` 字段 | S2 标签编辑区（添加/删除标签） |
| S1 按标签筛选 | `GET /api/identities/?tag=肩颈` 筛选参数 | S1 标签筛选器（标签云/chips） |
| S2 展示标签 | — | S2 客户名下方显示标签 badges |

### 不包含

| 功能 | 原因 |
|------|------|
| 永久删除客户 | 违反数据可追溯原则 |
| 自动标签（基于服务历史） | 需要规则引擎，Sprint-4+ |
| 标签颜色/分类系统 | 过度设计——自由文本即可 |
| 批量标签操作（选中多个客户打标） | Pilot 阶段客户量小，手动逐个操作即可 |
| 标签权限控制（谁能打标签） | 单店 Pilot，所有店员可操作 |
| 标签统计面板（哪些标签最常用） | Sprint-5+ |

## 5. Suggested Tags

### 门店常用标签（初始建议 12 个）

| # | 标签 | 用途 | 场景 |
|---|------|------|------|
| 1 | `肩颈` | 主要健康关注 | 与 S3 关注类别对应 |
| 2 | `腰背` | 主要健康关注 | |
| 3 | `疲劳` | 主要健康关注 | |
| 4 | `运动恢复` | 主要健康关注 | |
| 5 | `老客户` | 来过 3 次以上 | 回访优先级 |
| 6 | `新客户` | 首次到店 | 跟进关注 |
| 7 | `高意向` | 对服务很感兴趣，可能办卡/套餐 | 销售跟进 |
| 8 | `价格敏感` | 对价格比较在意 | 推荐策略 |
| 9 | `转介绍` | 朋友/家人介绍来的 | 来源追踪 |
| 10 | `需随访` | 标记需要重点随访 | 回访提醒 |
| 11 | `周末` | 习惯周末到店 | 排班规划 |
| 12 | `已流失` | 3 个月以上未到店 | 唤醒回访 |

**注意：** 这些是建议标签，不是系统预设。店员可以自由输入任何标签。系统不做枚举限制。

- 标签筛选器使用已有的 chip 样式（与 S3 类别选择一致）
- 点击标签 chip → 筛选结果
- 每个客户在列表中展示其标签
- "全部"为默认选中

### S2：客户总览页 — 标签 + 归档

```
┌─────────────────────────────────────────────────────┐
│ 张伟                        已激活                   │
│                                                     │
│ 标签： [肩颈] [老客户] [高意向] [+ 添加标签]         │
│                                                     │
│ [记录健康关注] [新建服务]                            │
│                                                     │
│ ... (健康档案、服务历史等) ...                       │
│                                                     │
│ ─────────────────────────────────────                │
│ [归档客户]   ← 底部操作区，浅色文字按钮              │
└─────────────────────────────────────────────────────┘
```

### 归档按钮位置

- 放在 S2 页面底部（不与主操作按钮竞争）
- 归档后显示"已归档"状态标签
- 已归档客户：显示 [恢复客户] 按钮

### 标签编辑位置

- 放在 S2 页面顶部，客户名和状态之间
- 点击 [+ 添加标签] → 弹出小输入框 → 输入标签名 → 回车添加
- 点击已有标签上的 × → 删除标签
- 不需要进入专门的编辑模式

### 归档客户在 S1 的显示

- 默认列表不包含已归档客户（`GET /api/identities/` 默认过滤 `status != archived`）
- 标签筛选器旁边增加 [包括已归档] 复选框
- 勾选后显示已归档客户（灰色显示）

## 7. Technical Notes

### API 扩展（最小修改）

**1. 客户恢复：**（新增 endpoint）

```
POST /api/identities/{id}/unarchive
→ activation_status: archived → active
→ Timeline entry: identity_unarchived
```

**2. 标签字段：**（扩展现有 endpoint）

`PATCH /api/identities/{id}` 已有。在 `IdentityUpdate` schema 中增加：

```python
class IdentityUpdate(BaseModel):
    display_name: str | None = None
    primary_store_id: uuid.UUID | None = None
    tags: list[str] | None = None  # NEW
```

同时在 `HealthIdentity` model 中增加：

```python
tags: Mapped[list[str]] = mapped_column(
    JSONB, nullable=False, default=list,
    comment="Customer tags for store-level categorization"
)
```

需要在 Platform DB migration 004 中添加 `tags` JSONB 列。

**3. 标签筛选：**（扩展现有 endpoint）

`GET /api/identities/` 已有。增加 query param：

```
GET /api/identities/?tag=肩颈
→ WHERE tags @> '["肩颈"]'  (PostgreSQL JSONB contains)
```

### 数据量预估

- MVP 阶段客户 < 100 人，标签总数 < 200 个
- JSONB 列性能足够（不需要独立 tags 表）
- 标签筛选使用 PostgreSQL GIN index on JSONB

## 8. Recommendation

### Priority

```
P1 — 建议在 Pilot 完成后、Sprint-4 或 Sprint-5 期间实施。
```

### 理由

- **不是 Pilot 阻塞项：** 当前 Pilot 验证的是闭环能不能跑通，标签和归档是"运营效率"问题，不是"功能完整"问题。
- **Pilot 数据验证需求：** 等 Pilot #1/#2 跑完后，确认：店员真的需要标签吗？哪些标签最常用？归档频率多高？
- **实施成本低：** 1 个新 endpoint + 1 个字段扩展 + 前端 3 处修改。预计 0.5-1 天。

### Pilot #2 收集的数据

在 PILOT-013 复盘时增加以下问题：

- "如果有标签功能，你最想打什么标签？"
- "你觉得归档有用吗？什么情况下你会归档一个客户？"
- "你现在怎么记住客户分类的？"

收集到真实反馈后再决定是否实施、实施哪些标签。
