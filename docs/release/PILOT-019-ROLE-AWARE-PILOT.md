## 1. Objective

验证 ROLE-002（首页角色分化）和 ROLE-003（按钮级权限控制）在真实门店多人多角色场景下是否按预期工作。

具体验证：

- 店长、健康管理师、服务人员打开系统后，是否看到符合各自角色的首页
- 服务人员是否被正确阻止执行新建客户、创建随访等超出权限的操作
- 店长是否能看到更偏管理视角的信息（全局客户数、激活数）
- 不同角色在同屏操作时，是否产生混淆或误操作

**不验证：** 后端 API 角色拦截（Sprint-5+）、观察者只读模式（Sprint-5）、归档/标签功能（PILOT-016 待定）。

## 4. Role-Based Test Scenarios

### Scenario A — 正常分工（Happy Path）

```
1. 店长登录 → 查看首页统计数据 → 进入 S1 查看全部客户列表
2. 健康管理师登录 → 查看待随访列表 → 选择一个待随访客户 → 完成 S2→S3→S4→S5→S6 闭环
3. 服务人员登录 → 搜索已有客户 → 完成 S4→S5（服务+反馈）→ S6 看到提示而非表单
```

**通过标准：** 三个角色各自完成路径，无不正确的按钮隐藏/显示。

### Scenario B — 服务人员越权测试（Negative Path）

```
1. 服务人员登录
2. 进入 S1 → 确认无"+ 新建客户"按钮
3. 进入 S6 → 确认无随访表单，仅显示提示文字
4. 尝试通过 URL 直接访问 /customers/new（如果前端路由存在）
   → 预期：前端不渲染表单（canCreate=false）
```

**通过标准：** 服务人员在 S1 和 S6 均被正确拦截，无法执行越权操作。

### Scenario C — 角色切换测试（同一设备）

```
1. 店长登录 → 完成一个客户闭环 → 退出
2. 服务人员在同一设备登录 → 确认首页变为 HealthAdvisorDashboard
3. 确认 S1 无"+ 新建客户"按钮（与店长登录时不同）
4. 退出 → 健康管理师登录 → 确认 S1 恢复"+ 新建客户"按钮
```

**通过标准：** 角色切换后页面状态完全按角色重置，无残留状态。

### Scenario D — 双角色同时操作（两台设备）

```
1. 设备 A：店长登录，进入 S1
2. 设备 B：服务人员登录，进入 S1
3. 店长在 S1 创建新客户（使用"+ 新建客户"）
4. 服务人员在 S1 搜索该新客户 → 点击进入 S2
5. 店长在 S6 为该客户创建随访
6. 服务人员在 S6 看到提示（无表单）
```

**通过标准：** 同一客户数据两个角色各见其应有界面，不互相影响。

## 5. What To Observe

### 5.1 首页第一印象

| 观察点 | 店长 | 健康管理师 | 服务人员 |
|--------|------|----------|---------|
| 打开系统后第一反应 | 是否注意到统计数据？ | 是否注意到待随访？ | 是否理解默认首页？ |
| 是否理解"为什么我的首页长这样" | 店长应感觉"我在看全局" | 健康管理师应感觉"我在看我的任务" | 服务人员可能疑惑"为什么我也是健康管理师工作台" |
| 是否需要解释首页差异 | 记录 | 记录 | 重点观察 |

### 5.2 按钮权限感知

| 观察点 | 说明 |
|--------|------|
| 服务人员是否尝试新建客户？ | 如果店员习惯性想点"新建客户"但找不到按钮，观察反应 |
| 服务人员是否尝试创建随访？ | 在 S6 看到提示文字后的反应——理解还是困惑？ |
| 服务人员是否感到"被限制"？ | 正面感受（"清晰知道该做什么"）还是负面感受（"功能被阉割"）？ |
| 非店长角色是否注意到"店员管理"缺失？ | 如果从未见过，可能不会注意到——这反而是好的 |

### 5.3 闭环完成情况

| 指标 | 店长 | 健康管理师 | 服务人员 | 合计 |
|------|------|----------|---------|------|
| 服务客户数 | | | | |
| S6 完成数 | | | | |
| S6 完成率 | | | | |
| 求助次数 | | | | |

### 5.4 服务人员 S6 提示的接受度

**关键观察：** 服务人员在 S5 保存反馈后进入 S6，看到"随访由店长或健康管理师创建"提示文字——

- 是否理解这句话的含义？
- 是否知道接下来该找谁（店长/健康管理师）来创建随访？
- 是否觉得流程"断了"？

## 7. Issue Logging Focus

### P0 — 阻塞角色验证

- 任何角色登录后首页白屏或报错
- JWT `role` 字段缺失或为空导致首页判断错误
- 服务人员可以越权创建客户或随访（前端校验失败）

### P1 — 角色体验问题

- 首页角色判断错误（店长看到健康管理师首页，或反之）
- 服务人员 S6 提示文字不清晰，导致店员不知道该做什么
- 服务人员因按钮隐藏而无法完成正常服务流程（S4/S5 被误伤）
- 角色切换后残留上一个角色的 UI 状态

### P2 — 文案/引导/体验

- 服务人员首页标题仍显示"健康管理师工作台"——可能混淆
- 店长首页统计数据与 S1 实际列表不一致
- 健康管理师待随访列表为空时显示不友好
- 任何角色对首页布局感到困惑

## 8. Success Criteria

### 必须满足（否则不通过）

- [ ] 三种角色登录后，首页各显示正确的 Dashboard 组件
- [ ] 服务人员在 S1 和 S6 被正确拦截，无法执行越权操作
- [ ] 店长和健康管理师可正常完成 S1→S6 完整闭环
- [ ] 服务人员可正常完成 S4→S5（服务+反馈），不被误伤
- [ ] 无 P0 问题
- [ ] 所有角色至少完成 1 个客户的对应流程

### 期望满足（通过但有优化空间）

- [ ] 服务人员理解 S6 提示，知道流程未断（知道该找谁创建随访）
- [ ] 店长认可首页统计信息对管理有帮助
- [ ] 健康管理师认可待随访列表比之前的通用首页更有用
- [ ] 无 P1 问题
- [ ] 店员无需口头解释即可理解角色差异

---

## 9. Stop Conditions

以下任一情况发生，立即停止试运行：

1. 任何角色登录后系统崩溃或白屏
2. 服务人员成功越权创建客户或随访（前端校验完全失效）
3. 角色判断逻辑错误导致店长无法使用正常功能
4. 首页统计数据与实际数据严重不一致
5. 同一设备角色切换后出现数据混淆（前一个角色的数据残留）
6. Founder 判断试运行不满足继续条件

## 10. Post-Pilot Decision Matrix

试运行完成后，根据结果进入以下决策路径：

| 结果 | 决策 |
|------|------|
| 全部 Must + 大部分 Should 通过 | ✅ 角色感知验证通过 → 进入持续 3-5 天扩大试运行（保持 3 角色） |
| 全部 Must 通过，Should 有 1-2 项未通过 | ⚠️ 有条件通过 → 修复 P1 问题后重新验证 |
| 有 Must 未通过 | ❌ 不通过 → 修复阻塞问题后重新执行 PILOT-019 |
| 服务人员 S6 提示引起严重流程断裂 | ⚠️ 需 Product 重新设计服务人员 S6 体验 |

## 12. Founder Decision Point

试运行完成后，Founder 需要决策：

```
角色感知验证结果：________

[ ] APPROVE — 角色差异正确，可进入持续 3-5 天扩大试运行
    → 扩大至 3 角色 × 3-5 天 × 10-20 客户
    → 进入 Sprint-4 规划（AI Summary + Knowledge + Dashboard）

[ ] CONDITIONAL — 角色差异基本正确，修复以下问题后重新验证：
    → 需修复：_________________________________

[ ] REJECT — 角色差异存在问题，需重新设计
    → 主要原因：_________________________________
```

**关键判断依据：**
1. PILOT-017 已验证单角色闭环可用性达标
2. PILOT-019 验证多角色共存时系统行为正确
3. 两个 Pilot 均通过 → 系统已准备好进入 Sprint-4（AI 集成）
