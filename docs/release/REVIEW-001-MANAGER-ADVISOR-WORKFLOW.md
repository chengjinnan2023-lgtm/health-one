## 1. Review Objective

对当前店长端与健康管理师端做联合工作流验收。验证：

1. 店长能否独立管理门店日常运营？
2. 健康管理师能否以自己负责的客户为中心完成日常工作？
3. 两个角色的协同是否形成闭环——店长分配 → 健康管理师接收 → 执行 → 反馈？

## 2. Manager Workflow

### 2.1 登录

```
店长打开 Health One
→ 输入 admin / health123
→ JWT role=店长
→ App.tsx:27 staff.role === "店长" → ManagerDashboard
→ 标题"店长工作台" + 欢迎语
```

**第一印象：** 店长立即感知到"这是我的专属工作台"。与健康管理师看到的页面完全不同。

### 2.2 店长首页（ManagerDashboard）

```
┌─────────────────────────────────────────┐
│ 店长工作台                               │
│ 张三，欢迎回来                           │
│                                         │
│ [今日新增 2] [今日服务 5] [待随访 4]      │
│                                         │
│ 客户结构                                 │
│ 总12 | 已激活8 | 待激活2 | 已归档2        │
│ 标签：肩颈(5) 老客户(3)                  │
│                                         │
│ 最近服务          │ 待随访               │
│ 张伟-健康舱 ✓     │ 张伟-📞电话-7/3      │
│ 李娜-咨询 …       │ 李娜-💬微信-7/4      │
│                   │ [查看全部 →]         │
│                                         │
│ 📥 导出日报（3 类 CSV）                   │
│                                         │
│ [客户管理]  [运营统计]                    │
└─────────────────────────────────────────┘
```

**店长在这里能回答的问题：**
- 今天忙不忙？（今日服务数）
- 有没有新客户？（今日新增）
- 有没有客户被遗忘？（待随访数）
- 客户结构健康吗？（总/已激活/待激活/已归档）
- 哪些标签最热门？（标签 Top 5）

**店长在这里能做的操作：**
- 点击任意客户名 → 进入 S2 客户总览
- 点击"查看全部"→ 进入完整待跟进队列
- 导出 3 类 CSV
- 跳转客户管理（S1）或运营统计

### 2.3 待跟进队列（/follow-up-queue）

```
共 6 个待跟进客户 · 随访计划 4 人 · 标签标记 2 人

张伟  [已激活]
📞 电话随访 · 负责人：张三 · 计划 2026/07/03
肩颈 · 老客户                        [查看客户 →]

李娜  [已激活]
🏷 高意向 · 负责人：李四 · 暂无随访计划
高意向 · 肩颈                        [查看客户 →]
```

**价值：** 店长在这里看到所有需要关注的客户——不管是通过随访计划标记的，还是通过标签标记的。两条信号源合并，去重，一眼看完。

**与健康管理师视角的差异：** 店长看到全店。健康管理师通过 `?staff_id=me` 只看自己的。

### 2.4 运营统计（/manager/stats）

```
运营统计
2026-06-29 ~ 2026-07-05

[本周] [本月]  ← 可切换

┌────────┬────────┬────────┐
│ 新增客户│ 服务记录│ 完成随访│
│   3    │   12   │   2    │
└────────┴────────┴────────┘

客户结构
总12 | 已激活8 | 待激活2 | 已归档2

标签 Top 5
肩颈(5) 老客户(3) …
```

**价值：** 本周/本月维度。比日报（今天）更宏观。用于周会、月度复盘。

**已知限制：** 无双周期对比（本周 vs 上周）。店长无法判断"3 个新客户是多还是少"。

### 2.5 导出日报

```
📥 导出日报
[📄 导出今日客户 CSV] [📄 导出今日服务 CSV] [📄 导出待随访 CSV]
```

店长点击 → `fetch()` 携带 JWT → 后端返回 UTF-8 BOM CSV → 浏览器下载。Excel/WPS 直接打开，中文不乱码。

**价值：** 数据离开系统的唯一通道。店长可发给总部/合作伙伴。

### 2.6 店员管理（/staff）

```
店员管理                     [+ 新增店员]
共 3 名店员

姓名     | 用户名   | 角色       | 状态 | 操作
张三     | admin   | 店长       | 正常 | [重置密码]
李四     | staff01 | 健康管理师  | 正常 | [停用] [重置密码]
服务人员  | staff02 | 服务人员   | 已停用 | [启用] [重置密码]
```

**价值：** 店长可自行管理店员——新增、停用、重置密码。不需要手动操作 SQLite。

**安全：** 后端强制 role 校验。健康管理师/服务人员访问 API 返回 403。无法停用自己。

### 2.7 分配客户 / 随访

**分配客户（S2）：**
```
负责人：[未指定 ▾]
        ├ 张三（店长）
        ├ 李四（健康管理师）  ← 店长选择
        └ 王五（服务人员）
```

店长打开客户 S2 → 下拉选择负责人 → PATCH identity。

**分配随访（S6）：**
```
分配给：[张三（店长）▾]
        ├ 张三（店长）
        ├ 李四（健康管理师）  ← 店长选择
        └ 王五（服务人员）
```

店长创建随访时 → 下拉选择分配给谁 → 创建随访。

**价值：** 店长在这里完成"管理"动作——把客户和随访分配给具体的健康管理师。这是角色协同的起点。

## 3. Health Advisor Workflow

### 3.1 登录

```
健康管理师打开 Health One
→ 输入 staff01 / pilot123
→ JWT role=健康管理师
→ App.tsx:28 role !== "店长" → HealthAdvisorDashboard
→ 标题"健康管理师工作台" + "李四，你有 3 个待随访客户"
```

**第一印象：** 健康管理师看到"我的待随访"和"我的客户"——以自己为中心的工作入口。

### 3.2 我的待随访

```
⏳ 我的待随访                          [查看全部 →]

张伟 · 电话 · 7/3
李娜 · 到店 · 7/4
王五 · 微信 · 7/5
```

数据来源：`GET /api/dashboard/follow-up-queue?staff_id=me`（1 次请求，非 N+1）。

**价值：** 健康管理师一眼看到"我今天/本周该随访谁"。不再淹没在全店数据中。

对比 FEATURE-007 之前：健康管理师首页显示的是全店前 5 个活跃客户的待随访（N+1 查询，最多 15 次请求）。现在只显示分配给自己的。

### 3.3 我的客户（/my-customers）

```
我的客户
共 5 名负责客户

[搜索客户名…]

张伟  [已激活]
肩颈 · 老客户
⏳ 待随访：电话 · 2026/07/03            [查看 →]

李娜  [已激活]
高意向                                   [查看 →]

王五  [待激活]
新客户                                   [查看 →]
```

数据来源：`GET /api/identities/?assigned_staff_id=me` + `GET /api/dashboard/follow-up-queue?staff_id=me`（并行 2 次请求）。

**价值：** 健康管理师看到"谁是我的客户"。有待随访的客户有橙色 ⏳ 提示——不用点进去就知道该做什么。

**空状态：** 如果店长还没分配客户 → "暂无负责的客户。请联系店长为客户分配负责人。"

### 3.4 进入客户总览（S2）

健康管理师点击"查看 →" → 进入 S2。

```
张伟                            [已激活]
负责人：李四（健康管理师）         ← 仅显示，不可改

标签：肩颈 · 老客户               ← 可编辑

健康档案 | 服务历史
随访     | 最近动态
```

**与店长 S2 的差异：** 健康管理师看不到负责人变更下拉（仅店长可见）。标签可正常编辑。其余完全相同。

### 3.5 完成服务闭环

```
S2 → [新建服务] → S4 服务记录 → 保存
    → [继续反馈] → S5 反馈记录 → 保存反馈
    → [创建随访] → S6 随访 → 选择方式/时间/分配给 → 创建
```

S6 的"分配给"下拉：
- 店长使用 S6 时：下拉列出全店店员，默认选中客户当前负责人
- 健康管理师使用 S6 时：当前为 disabled input（显示自己名字），随访默认分配给自己

**已知限制：** 健康管理师的 S6 分配下拉未获取店员列表（仅 isManager 时 fetch）。健康管理师创建随访时只能分配给自己。如需分配给其他人，需店长操作。

## 4. Cross-Role Hand-off

### 4.1 完整协同流程

```
店长                            健康管理师
│                               │
├─ S2：分配客户给李四             │
│   PATCH assigned_staff_id      │
│                               │
│                               ├─ 首页刷新 → "共 5 名负责客户"
│                               ├─ /my-customers → 张伟出现
│                               │
├─ S6：为张伟创建随访             │
│   分配给：李四                  │
│                               │
│                               ├─ 首页刷新 → 待随访 +1
│                               ├─ ⏳ 我的待随访 → 张伟出现
│                               │
│                               ├─ 点击 → S2 张伟
│                               ├─ S4 记录服务 → 保存
│                               ├─ S5 记录反馈 → 保存
│                               ├─ S6 创建/完成随访
│                               │
│                               ├─ 首页刷新 → 待随访 -1
│                               │
├─ /follow-up-queue → 张伟消失   │
│   （随访已完成）                │
│                               │
├─ /manager/stats → 完成随访 +1  │
└─ 导出 CSV（如需）              └─ 继续下一位客户
```

### 4.2 关键交接点

| 交接点 | 店长动作 | 健康管理师反应 | 实现状态 |
|--------|---------|-------------|---------|
| 客户分配 | S2 下拉选择负责人 | /my-customers 出现该客户 | ✅ |
| 随访分配 | S6 下拉选择分配给 | 首页"我的待随访"出现该随访 | ✅ |
| 随访完成 | 无需动作 | 首页待随访自动减少（下次刷新） | ✅ |
| 客户停用 | 店员管理 → 停用 | 停用店员已有的分配不受影响 | ⚠️ 见 §6 |

### 4.3 当前协同的边界

```
✅ 已实现：
  店长分配客户 → 健康管理师可见
  店长分配随访 → 健康管理师可见
  健康管理师完成随访 → 全店队列更新

⚠️ 未实现（但当前阶段可接受）：
  健康管理师无法"拒绝"或"转交"分配 → 口头沟通
  分配变更无通知 → 刷新页面即可
  店长无法查看健康管理师工作量 → ManagerStats 无 staff 维度
```

## 6. What Still Feels Missing

按对日常运营的影响排序：

| # | 缺失 | 影响 | 谁受影响 |
|---|------|------|---------|
| 1 | **健康管理师 S6 不能分配给别人** | 健康管理师创建随访只能分配给自己。如果想转给另一个健康管理师，需要店长操作 | 健康管理师 |
| 2 | **店长看不了店员工作量** | ManagerStats 有全店数据，但无 staff 维度——不知道"李四本周服务了多少客户" | 店长 |
| 3 | **无通知/提醒** | 店长分配客户后，健康管理师不会收到通知——需主动刷新首页 | 双方 |
| 4 | **停用店员后无清理提示** | 停用店员时，系统不提示"该店员还有 N 个负责客户/N 个待随访" | 店长 |
| 5 | **无双周期对比** | 运营统计无"本周 vs 上周"——店长不知道趋势 | 店长 |
| 6 | **健康管理师主页无"今日服务"数** | 健康管理师首页看不到自己今天的服务数量 | 健康管理师 |

## 8. Recommendation

### 8.1 店长端是否已具备基本门店管理可用性？

```
YES — 店长可以独立完成"看全局 → 找问题 → 分配工作 → 管理店员 → 导出数据"的完整管理闭环。
```

店长有 5 个独立页面（首页、待跟进队列、运营统计、店员管理、客户管理）+ 日报导出。覆盖日常管理核心场景。

### 8.2 健康管理师端是否已具备基本日常执行可用性？

```
YES — 健康管理师可以"看我的待随访 → 看我的客户 → 进入客户 → 完成服务 → 记录反馈 → 创建随访"。
```

从 FEATURE-007 开始，健康管理师终于有了"自己的"视图——不再是全店数据的旁观者。FEATURE-008 的"我的客户"让这个闭环更完整。

**但有一处明显摩擦：** 健康管理师 S6 只能自分配。如果想把随访转给另一位健康管理师，必须找店长操作。考虑在下一轮小优化中修复（给健康管理师也 fetch 店员列表）。

### 8.3 角色协同是否已形成闭环？

```
YES — 店长分配 → 健康管理师接收 → 执行 → 反馈 → 队列更新。闭环存在且可运转。
```

关键交接点（客户分配、随访分配）均即时生效——健康管理师刷新首页即可看到新分配。

### 8.4 下一步最值得做的是？

```
建议：扩大试运行 > 补一轮小优化 > 开新功能
```

**理由：**

当前系统已实现 8 个 Feature，两个角色的基本工作流均已可用。接下来最需要的是**真实门店反馈**——在真实使用中暴露以下问题：

1. 健康管理师是否觉得"我的客户"有用？
2. 店长是否真的用"待跟进队列"发现遗漏？
3. 店员管理在实际使用中是否有摩擦？
4. 日报导出能满足真实的汇报需求吗？

**建议的扩大试运行方案：**

| 维度 | 建议 |
|------|------|
| 门店 | STORE-001（不变） |
| 角色 | 店长 1 人 + 健康管理师 1-2 人 + 服务人员 1 人 |
| 天数 | 连续 5 天 |
| 客户 | 目标 10-20 名真实客户完成闭环 |
| 设备 | 至少 2 台（店长 1 台 + 健康管理师 1 台） |
| 观察重点 | 协作是否流畅、健康管理师是否使用"我的客户"、队列是否帮助发现遗漏 |

**试运行前建议完成的小优化（P1-1）：**
健康管理师 S6 允许分配给本店其他店员（30 分钟改动——让健康管理师也 fetch 店员列表）。
